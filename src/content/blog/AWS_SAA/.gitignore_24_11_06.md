**솔루션 설계자가 매일 한 번 실행되고 완료하는 데 최대 2 시간이 걸리는 데이터 처리
작업을 만들고 있습니다. 작업이 중단되면 처음부터 다시 시작해야 합니다.
솔루션 설계자가 가장 비용 효율적인 방식으로 이 문제를 해결하려면 어떻게 해야 하나요?**

<div class="options">

A. 크론 작업에 의해 트리거되는 Amazon EC2 예약 인스턴스에서 로컬로 실행되는 스크립트를 만듭니다.

B. Amazon EventBridge 예약 이벤트에 의해 트리거되는 AWS Lambda 함수를 생성합니다.

C. Amazon EventBridge 예약 이벤트에 의해 트리거되는 Amazon ECS(Amazon Elastic
Container Service) Fargate 작업을 사용합니다.

D. Amazon EventBridge 예약 이벤트에 의해 트리거된 Amazon EC2 에서 실행되는 Amazon ECS(Amazon Elastic Container Service) 작업을 사용합니다.</div>

        <span class="answer">C</span> <br/>

##### 1. **비용 효율성**

- Fargate는 **서버리스 컴퓨팅 방식**으로, 인프라 관리 없이 컨테이너 단위로 비용을 지불함. 컨테이너가 실행되는 시간에만 비용이 발생하므로, 매일 2시간 동안만 필요한 작업에 대해 최적의 비용 효율성을 제공함.
- 서버를 항상 켜 두지 않고 작업이 끝나면 자동으로 종료되므로, 추가적인 유지 관리 비용이 없음.

##### 2. **신뢰성과 내구성**

- Fargate는 작업이 중단될 경우 **자동으로 복구**하거나 다시 시작할 수 있도록 설계됨. 작업이 중단되더라도 처음부터 재실행하는 비용 부담이 줄어들며, Fargate에서 안정적으로 다시 시작할 수 있음.
- Amazon ECS가 작업 상태를 모니터링하고, 오류가 발생하면 자동으로 복구할 수 있도록 지원함.

##### 3. **확장성 및 유연한 리소스 할당**

- Fargate는 작업에 필요한 CPU와 메모리를 유연하게 할당할 수 있음. 매일 2시간 동안의 작업이기 때문에, 필요한 리소스를 최소한으로 맞춤 설정하여 비용을 절감할 수 있음.
- 이 작업은 특정 스케줄에 따라 매일 반복되므로, Fargate를 통해 효율적으로 실행되도록 관리할 수 있음.

##### 4. **운영 오버헤드 감소**

- Fargate를 사용하면 인프라 프로비저닝이나 서버 유지보수를 하지 않아도 됨. 솔루션 설계자는 오로지 데이터 처리 작업에 집중할 수 있고, 서버 관리 부담 없이 비용 절감이 가능함.

애플리케이션이 필요합니다. 애플리케이션은 데이터 엔터티 간의 관계를 분석하고
사용자에게 권장 사항을 제공해야 합니다.
최소한의 운영 오버헤드로 이러한 요구 사항을 충족하는 솔루션은 무엇입니까?**

**한 회사에서 대량의 데이터를 저장할 새로운 애플리케이션을 만들고 있습니다. 데이터는
매시간 분석되며 여러 가용 영역에 배포된 여러 Amazon EC2 Linux 인스턴스에 의해
수정됩니다. 필요한 저장 공간의 양은 향후 6 개월 동안 계속 증가할 것입니다.
솔루션 설계자는 이러한 요구 사항을 충족하기 위해 어떤 스토리지 솔루션을 권장해야
합니까?**

<div class="options">

A. Amazon S3 Glacier 에 데이터를 저장합니다. 애플리케이션 인스턴스에 대한 액세스를
허용하도록 S3 Glacier 볼트 정책을 업데이트합니다.

B. Amazon Elastic Block Store(Amazon EBS) 볼륨에 데이터를 저장합니다. 애플리케이션
인스턴스에 EBS 볼륨을 탑재합니다.

C. Amazon Elastic File System(Amazon EFS) 파일 시스템에 데이터를 저장합니다.
애플리케이션 인스턴스에 파일 시스템을 마운트합니다.

D. 애플리케이션 인스턴스 간에 공유되는 Amazon Elastic Block Store(Amazon EBS)
프로비저닝된 IOPS 볼륨에 데이터를 저장합니다.</div>

<details class="markdown-toggle">
  <summary><b>Answer</b></summary>
  <div markdown="1">
    <ul>
      <li>
        <span class="answer">C</span> <br/>
      </li>
    </ul>
    
  - **확장성**
    - Amazon EFS는 **완전 관리형 확장 파일 스토리지**로, 데이터를 추가할수록 자동으로 용량이 늘어남. 따라서 **6개월 동안 지속적으로 증가하는 저장 공간 요구**를 문제없이 충족할 수 있음.
    - EFS는 무제한 확장이 가능하여 수백 테라바이트 이상의 데이터를 저장할 수 있음.
- **동시 액세스 지원**
    - 여러 EC2 인스턴스에서 동시에 EFS 파일 시스템을 마운트할 수 있으며, 파일 시스템은 여러 **가용 영역(AZ)에 걸쳐 분산**되어 있어 고가용성을 제공함.
    - 애플리케이션이 다수의 EC2 인스턴스를 통해 데이터를 수정하므로, **파일 시스템을 여러 인스턴스가 동시에 읽고 쓸 수 있는 환경**이 필요함. EFS는 NFS(Network File System) 프로토콜을 지원하여 여러 인스턴스가 동시에 파일을 안전하게 사용할 수 있음.
- **고가용성과 내구성**
    - EFS는 AWS의 다중 AZ 인프라를 기반으로 하여 **고가용성**과 **99.999999999%의 내구성**을 보장함. 데이터가 여러 가용 영역에 걸쳐 저장되므로, 인스턴스가 다른 AZ에 배포되더라도 동일한 데이터를 안정적으로 접근할 수 있음.
    - 따라서 데이터가 분산되어 있어, 하나의 AZ에 문제가 생겨도 데이터 손실 없이 서비스를 지속할 수 있음.
- **운영 오버헤드 최소화**
    - Amazon EFS는 관리형 파일 스토리지이므로, 수동으로 인프라를 관리하거나 파일 시스템을 확장할 필요가 없음. EFS가 자동으로 확장과 축소를 처리해주므로, 운영에 필요한 리소스와 시간을 절약할 수 있음.
    - 또한 EC2 인스턴스에서 간단히 파일 시스템을 마운트하는 방식으로 사용 가능하므로 별도의 스토리지 관리 없이 데이터 접근이 가능함.
  </div>
</details>

**회사는 PostgreSQL 다중 AZ DB 인스턴스용 Amazon RDS 에 데이터를 저장하는
애플리케이션을 관리합니다. 트래픽 증가로 인해 성능 문제가 발생합니다. 회사에서는
데이터베이스 쿼리가 성능 저하의 주요 원인이라고 판단합니다.
솔루션 아키텍트는 애플리케이션 성능을 향상시키기 위해 무엇을 해야 합니까?**

<div class="options">

A. 다중 AZ 대기 복제본에서 읽기 트래픽을 제공합니다.

B. Transfer Acceleration 을 사용하도록 DB 인스턴스를 구성합니다.

C. 원본 DB 인스턴스에서 읽기 전용 복제본을 생성합니다. 읽기 복제본에서 읽기 트래픽을
제공합니다.

D. 애플리케이션과 Amazon RDS 사이에 Amazon Kinesis Data Firehose 를 사용하여
데이터베이스 요청의 동시성을 높입니다.</div>

<details class="markdown-toggle">
  <summary><b>Answer</b></summary>
  <div markdown="1">
    <ul>
      <li>
        <span class="answer">C</span> <br/>
      </li>
    </ul>
    
- **읽기 트래픽 분산**
    - 애플리케이션에서 발생하는 대부분의 성능 저하는 주로 **읽기 트래픽 증가**에 의해 발생할 수 있음. 읽기 전용 복제본을 사용하면 **읽기 트래픽을 복제본으로 분산**하여 원본 DB의 부하를 줄일 수 있음.
    - 복제본에서 읽기 요청을 처리하도록 구성하면, 원본 DB 인스턴스는 쓰기 작업에 더 많은 리소스를 집중할 수 있어 전반적인 성능이 개선됨.
- **복제본 동기화**
    - Amazon RDS의 읽기 복제본은 **자동으로 원본 DB와 동기화**되며, 데이터 변경 사항이 실시간으로 반영됨. 이를 통해 읽기 복제본에서 최신 데이터를 조회할 수 있어 데이터 일관성을 유지함.
    - 동기화가 이루어지는 동안에도 읽기 트래픽을 처리하므로 **데이터 정합성을 유지하면서도 높은 성능을 제공**할 수 있음.
- **비용 효율성**
    - 읽기 복제본은 원본 DB와 동일한 데이터 구조를 유지하지만, 읽기 요청에만 사용되므로 리소스를 효율적으로 사용할 수 있음. 별도의 고성능 인스턴스를 추가하지 않고도 복제본을 통해 성능을 개선할 수 있음.
    - 트래픽 증가에 따라 복제본을 추가하거나 제거할 수 있어, 비용 효율적으로 확장 가능함.
- **고가용성 및 확장성**
    - Amazon RDS는 다중 AZ에 걸쳐 복제본을 배포할 수 있어 가용성과 복원력이 높음. **복제본이 여러 AZ에 걸쳐 생성**되면 특정 AZ에서 문제가 발생하더라도 복제본을 통해 지속적으로 읽기 요청을 처리할 수 있음.
    - 읽기 복제본을 필요에 따라 여러 개 생성하여 읽기 성능을 수평 확장할 수 있음.
  </div>
</details>

**한 회사에서 다양한 기계에서 매일 10GB 의 원격 분석 데이터를 수집합니다. 이 회사는
소스 데이터 계정의 Amazon S3 버킷에 데이터를 저장합니다.
이 회사는 이 데이터를 분석에 사용하기 위해 여러 컨설팅 기관을 고용했습니다. 각
대행사는 분석가를 위해 데이터에 대한 읽기 액세스 권한이 필요합니다. 회사는 보안과
운영 효율성을 극대화하는 솔루션을 선택하여 소스 데이터 계정의 데이터를 공유해야
합니다.
이러한 요구 사항을 충족하는 솔루션은 무엇인가요?**

<div class="options">

A. 각 기관의 데이터를 복제하도록 S3 글로벌 테이블을 구성합니다.

B. S3 버킷을 제한된 시간 동안 공개합니다. 에이전시에게만 알립니다.

C. 대행사가 소유한 계정에 대한 S3 버킷의 교차 계정 액세스를 구성합니다.

D. 소스 데이터 계정의 각 분석가에 대해 IAM 사용자를 설정합니다. 각 사용자에게 S3
버킷에 대한 액세스 권한을 부여합니다.</div>

<details class="markdown-toggle">
  <summary><b>Answer</b></summary>
  <div markdown="1">
    <ul>
      <li>
        <span class="answer">C</span> <br/>
      </li>
    </ul>
    
**옵션 C**인 *대행사 소유의 계정에 대한 S3 버킷 교차 계정 액세스를 구성하는 방식*이 적합한 이유는 **보안과 운영 효율성을 극대화하면서도, 분석가들이 독립적으로 접근할 수 있는 환경을 제공**할 수 있기 때문임.

##### 이유

1. **보안 관리의 단순화**

- 교차 계정 액세스는 **IAM 역할을 통해 접근을 제어**하므로, 회사는 개별 IAM 사용자나 각 분석가의 자격 증명을 직접 관리하지 않아도 됨.
- 대행사의 계정을 S3 버킷에 대해 교차 계정 액세스로 구성하면, 대행사 계정 내의 IAM 역할을 사용해 **필요한 읽기 권한만 부여**할 수 있음. 이로 인해 보안과 접근 제어가 간단해지고, 자격 증명 관리 부담이 줄어듦.

2. **각 대행사의 독립적인 사용자 관리**

- 교차 계정 액세스를 통해 각 대행사는 **자체 계정에서 권한을 관리**할 수 있으므로, 회사가 개별 분석가마다 IAM 사용자를 생성할 필요가 없음.
- 대행사에서 사용자 관리를 자체적으로 진행하므로, 회사는 대행사의 IAM 사용자에 대한 관리 부담 없이 **S3 버킷에 대한 접근 권한을 간단히 제공**할 수 있음.

3. **확장성 및 효율성**

- 새로운 대행사가 추가되거나 기존 대행사의 인원이 변경될 경우에도 **S3 버킷 정책만 업데이트하면** 됨. 개별 사용자를 추가하거나 제거하는 작업 없이 각 대행사의 IAM 역할을 통해 관리가 가능하므로 운영 효율성이 높음.

    - SQS는 **내구성이 높은 메시지 대기열**을 제공하여, Lambda 함수가 정상적으로 처리하지 못한 메시지를 대기열에 유지함. Lambda 함수에서 일시적인 오류가 발생해도 SQS가 **메시지를 유실 없이 보관**하고 재시도할 수 있게 하여, 안정적인 이벤트 처리가 가능함.
    - SNS는 한 번 전달된 메시지를 보관하지 않지만, SQS는 메시지를 저장하므로 안정성이 더 높음.
- **비동기 처리와 병렬성**
    - SQS는 비동기적으로 작동하므로, Lambda 함수는 **이벤트가 발생할 때마다 대기열에서 메시지를 읽어 비동기로 처리**할 수 있음. 이를 통해 병렬로 Lambda 함수를 실행하여 효율적으로 이벤트를 처리함.
    - Lambda 함수가 필요한 만큼 인스턴스를 생성하여 병렬로 작업할 수 있어, 여러 이벤트를 동시에 처리해야 하는 경우 유리함.
- **운영 오버헤드 감소**
    - SQS는 **서버리스로 관리되므로 운영 오버헤드가 적고 비용 효율적**임. 개발팀은 SQS 대기열을 설정해놓고 Lambda와 자동으로 연결하여 이벤트를 처리할 수 있어 운영이 간단해짐.

##### 역할(Role)

- 역할은 **AWS 리소스나 사용자에게 특정 권한을 부여하기 위해 생성된 권한 그룹**임.
- 예를 들어 Lambda 함수가 S3 또는 KMS 키에 접근하려면 **Lambda 함수에 부여할 역할을 생성하고 이 역할에 적절한 권한을 추가**해야 함.
- 역할을 Lambda 함수에 연결하면 Lambda가 지정된 권한을 사용해 다른 리소스에 접근할 수 있음.
- Lambda 함수가 KMS 키를 사용하는 경우, 이 역할에 **KMS 키에 대한 암호 해독 권한**이 필요함.

##### 정책(Policy)

- 정책은 **AWS 리소스에 대한 구체적인 허가를 정의**하는 JSON 문서임. 정책을 통해 어떤 리소스에 대해 어떤 작업을 수행할 수 있는지를 지정할 수 있음.
- 정책은 역할이나 사용자에 **세부 권한을 설정**하는 방식으로, 예를 들어 Lambda의 역할에 특정 KMS 키의 암호 해독 권한을 부여하는 정책을 추가할 수 있음.
- **정책을 통해서만 권한을 구성할 수 있기 때문에**, 역할을 만든 후에도 정책을 추가로 설정해야 함.

1. **B. KMS 키의 정책에서 Lambda IAM 역할에 대한 암호 해독 권한을 부여**
   - KMS 키의 정책에서 Lambda 역할을 명시적으로 정의해 **Lambda 함수가 특정 KMS 키에 접근할 수 있도록 허가**함.
   - Lambda가 해당 키를 사용해 암호화된 파일을 해독할 수 있게 하는 필수 단계임.
2. **E. kms:암호 해독 권한이 있는 새 IAM 역할을 만들고 실행 역할을 Lambda 함수에 연결**
   - Lambda 함수가 S3에서 파일을 가져와 KMS 키로 암호 해독하려면 이 역할이 필요함.
   - 새 IAM 역할을 생성하고 `kms:Decrypt`와 같은 권한을 추가하여 Lambda가 이 역할을 통해 **KMS 키에 접근하여 파일을 해독할 수 있는 권한을 갖도록 설정**함.

**"kms:암호 해독" 권한을 가진 새 IAM 정책을 Lambda 함수에 직접 첨부하는 방식**이 적합하지 않은 이유는, Lambda 함수가 직접 정책을 첨부할 수 있는 개별 사용자나 그룹이 아니기 때문임. 대신, Lambda 함수는 역할(Role)을 통해 권한을 위임받아 실행해야 함.

1. **일반(퍼블릭) 인증서 필요성**
   - 일반 인증서(퍼블릭 인증서)는 웹사이트 방문자들이 웹 브라우저를 통해 **공개적으로 접근하는 트래픽을 암호화**하는 데 적합함. 국제적인 회사의 하위 도메인은 외부 고객 및 사용자들이 접근할 수 있는 **퍼블릭 웹사이트**이므로 퍼블릭 인증서가 요구됨.
   - **AWS ACM 퍼블릭 인증서**는 AWS에서 관리되며 무료로 제공되고, 브라우저와 디바이스에서 신뢰할 수 있는 인증서로 발급됨.
2. **비공개 인증서 사용 제한**
   - 비공개 인증서는 회사 내부망 등 **내부 시스템**의 트래픽 암호화에 주로 사용되며, 퍼블릭 인터넷을 통해 접근하는 사이트에는 적합하지 않음.
   - 퍼블릭 브라우저에서 비공개 인증서는 신뢰되지 않아 **보안 경고가 표시될 수 있음**.
3. **AWS ACM 와일드카드 인증서 활용**
   - `.example.com` 와일드카드 인증서를 통해 각 국가별 하위 도메인(`country1.example.com`, `country2.example.com` 등)을 포함한 모든 하위 도메인을 하나의 인증서로 암호화할 수 있음.
   - **AWS ACM**을 통해 관리되므로 갱신이 자동화되고, 로드 밸런서에서 쉽게 사용 가능함.

##### 도메인 소유권 유효성 검사

- **DNS 레코드 추가**는 AWS ACM에서 인증서를 발급하기 전 도메인의 소유권을 확인하는 절차임.
- 소유권 검증은 **AWS가 도메인을 신뢰할 수 있는지 확인하는 과정**이며, DNS 공급업체에 해당 도메인의 소유임을 증명할 수 있는 레코드를 추가하는 방식으로 수행됨.
